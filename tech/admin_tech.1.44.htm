Optio Trouble Shooting
To troubleshoot Optio:
1
First, verify that the Prelude Integration is setup correctly. Is the document flagged to print in the Optio format? Is the document flagged to fax in the Optio format? Is the “printer form queue” set up for Optio? Check /FM and /OPTIO.MNT.
2
If those are correct, check the print streaming the Optio Virtual printer. Log in as an administrator and disable the Optio Virtual printer:
-
Live Account – adsopt
-
Test Account – testopt
3
Type disable <virtual printer name>
4
Submit the job from Prelude and note the job number.
5
List the Virtual Printer queue and find your job number: qcheck –L –P <virtual printer name>
6
Check the print stream stored in /var/spool/qdaemon/<filename>. You can edit this entry with many programs, but you should know what is on each line in this file. Things to look for:
-
Does the first line have the Optio header line? This line passes administrative information to Optio (e.g., actual Unix printer device, form number being processed, page type, language number). Each page must have this line followed by 66 lines of Prelude document.
-
Does the document have 66 lines for every page? Must be exact, no over or under allowed.
-
Is there any programming message inside this document (e.g., un-initialized variables).
7
If the print stream has made it this far, you can test the document in Optio. Turn on logs in the etc directory of Optio. The logs capture error messages from Optio. The file is usually in /usr/odcs/etc and called odcs.ini. Here is an example of the TRACEMASK line to change to turn the logs on and off.
-
Logs off:
[Log Defaults]
WARNINGMASK="None"
TRACEMASK="None"
-
Logs on:
[Log Defaults]
WARNINGMASK="None"
TRACEMASK="All -Elements"
8
While the logs are on, Optio generates a Trace.log, which allows you to see what happened in Optio. You can verify that the document was mapped correctly, the output drawn as expected, and was sent to the correct device. Fatal messages are stored in the Fatal.log file. Logs are stored in /usr/odcs/errors.
9
Disable the actual Unix printer device and enable the Optio Virtual printer:
-
disable <Unix printer device>
-
enable <virtual printer name>
10
Now that the printer device is disabled and the Optio Virtual printer enabled, look for the job number in the disabled Unix printer device. Use the lpstat –p<printer device> command.
-
If the job doesn’t show up, Optio had problem with the document. Look at the errors in the logs.
-
If the job did show up in the printer, enable the printer device.
11
Depending on the errors in the print stream or the Optio logs, you may need help from Prelude.
Note: Turn off the logs immediately or your /var file system will fill up. If that happens, no jobs can be spooled, Optio or not! You can remove the log files in the errors directory to regain space if necessary. The logs are always appended to or created.
Optio Log Types
•
Warning.log - Warning messages from the ocserver command - not fatal - document will continue, but sections of document may or may not be included. Usually fonts, images, etc.
•
Fatal.log - Fatal message from the ocserver command. Document cannot be completed with a reason in this log. Usually, missing components such as DCL, map, segment, format, channel, function, or structure within these files.
•
Trace.log - Trace log shows Optio code from start to finish of the documents. Shows mapped variables, logical steps taken within the DCL, all assignments etc.
Up to 50 megabytes in size - Don’t leave logs on!

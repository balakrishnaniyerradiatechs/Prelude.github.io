Background Jobs
There are times when running a job from a terminal is not convenient. The terminal is tied up the entire time it takes the job to run, which could be hours, and the terminal cannot be logged off. Fortunately, jobs can be set up to run in the background, or as phantoms, which allows the jobs to run without being associated to a physical terminal. This frees the terminal and the user to go on to other tasks. Accommodations must be made to handle any screen input or output the background job is expecting, since the job will not be interacting with a screen.
Background Jobs in Unix
A user can start an unlimited number of background jobs, keeping in mind the capabilities of the processor and the job load that day. Each background job has a job number unique to the user and a process ID unique to the system. Use the “jobs” command to list the current background and stopped jobs with their job numbers and process IDs. You can use these numbers in Unix job control commands. Remember that a Unix background job cannot take input from the terminal but will send standard output to the terminal if not redirected. You can redirect input to be read from a file and redirect standard output and error to be saved in a file.
Example: cleanup.my.directory > my.output 2>my.errors &
The ampersand (&) says run the job “cleanup.my.directory” in the background, sending the output (>) to a file called “my.output” and any errors during runtime (2>) to a file called “my.errors”.
A job currently running on the terminal can be stopped by hitting CTRL “z” and then resumed in the background by typing “bg<return>”. Note: In some cases, Prelude has chosen an STTY setting that will require a CTRL “a” to stop the job rather than a CTRL “z”.
Background Jobs in UniData
The “PHANTOM” verb executes background jobs in UniData. All output is stored in the _PH_ file with a record ID with the user’s login name, the internal time, and the process ID. An “INPUT” aborts the job if data was not provided. The “PHANTOM” command executes an existing login paragraph in the VOC as part of its normal processing. However, any interactive-only processing in the login paragraph may be conditionally bypassed by checking @USER.TYPE for phantom users.
Example: PHANTOM LIST ORDER
The UniQuery statement “LIST ORDER” runs in the background and the output is stored in _PH_. You can use the verb “CHECK.PHANTOM” to invoke SB+ as it submits the background job (similar to “PHANTOM” in PICK).

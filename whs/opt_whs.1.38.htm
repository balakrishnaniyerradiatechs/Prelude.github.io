Cycle Counting
The cycle count system performs periodic physical counts at the bin level a given number of times per year. A nightly process generates queue records with the appropriate number of bins to count in a given day. The radio frequency cycle count entry then directs the counter on what to count and records the counts entered. The goal is to undertake physical counts on a daily basis rather than performing annual or semi-annual counts of the entire warehouse. Emergency cycle counts are created when the system detects a potential error such as a negative bin quantity or a change during picking. The maintenance flags and procedures for cycle counts and emergency cycle counts are included in this document. Refer to Cycle Count Options for more information on setting up your warehouse for Cycle Counting.

Cost / Tolerance Maintenance
Path: Inventory > Physical Inventory > Cycle Counting Procedure > Cost/Tolerance Maintenance (/CYC.COUNT.TOL.MNT)
This maintenance routine defines tolerance levels for different cost ranges. Tolerance levels may be defined by unit cost or by bin value. These tolerances are used in the 1st level cycle count entry to determine if a product needs a second count performed. This is discussed in more detail in the 1st level cycle count entry sections. Ranges of costs are entered in this routine and an associated tolerance percentage for each range.
•
Cycle Count Type: Indicates if the unit cost or the total bin value is used in determining variances.
-
C - (or null) Variances are calculated on the unit cost. This means that a one-piece unit cost is used to determine the variance. For example, using the above table, if a product with a cost of 122.95 is off by 10% or more it is placed into the second level count queue.
-
B - Variances are calculated on the bin value. This means that the system looks at a bin as a unit even though it could be made up of 100 pieces. For example, a bin could contain 100 pieces of an item costing $5.00, 5 pieces of an item costing $100.00, or 1 piece of an item costing $500.00. Whichever the case, only a variance of less than 5% would be allowed according to the above table. The bin value is determined by multiplying the product unit cost by the quantity in the bin. This method is useful if there are a lot of low priced products that are stocked in large quantities.
-
For example: a bin contains 100 pieces of an item costing $5.00. With the unit cost method, a second count is required when the count is off by 15% (15 pieces) or more. With the bin value method, a second count is required when the count is off by 5% (5 pieces) or more.
•
From Cost: Defines the beginning unit cost or bin value for the range.
•
To Cost: Defines the ending unit cost or bin value for the range.
•
Tolerance%: Defines the tolerance % for the cost range. If the count is off by this percent or more, then a second count is required.

•

-

-

Monthly / Nightly Processing
Cycle Count Monthly Processing
Path: Inventory > Physical Inventory > Cycle Counting Procedure > Cycle Count Monthly Processing (/CYC.COUNT.MONTHLY.PROCESS)
This routine is responsible for updating a control record for each warehouse with bin count frequencies, number of bins per bin count frequency, number of daily bin counts per bin count frequency, and the list of bins per bin count frequency. Remember that only pick, bulk, and retail bins are selected for cycle counting.
There are two main purposes for running this monthly process:
•
A cycle count control file is populated with the number of daily bin counts per bin count frequency. In other words, monthly processing determines the number of bins that need to be counted per day for each frequency. The formula used for this is:
-
Number of Bins in a Given Count Frequency * Frequency
-
Number of Count Days Per Warehouse
-
If a zone has a bin count frequency of 4 (meaning all bins in that cycle count zone should be counted four times for the year). The total number of bins in that zone equals 150, and the warehouse counts bins 200 days per year. The system would calculate the number of daily counts for that zone to be 3.
•
Bins that need to be scheduled for counting are set up in a special control file used by the cycle count nightly process. Monthly processing becomes an important process to run consistently in the case of bin location additions and/or deletions. Each time this process is run, the system selects the appropriate bins and sorts them by warehouse, bin count frequency, cycle count zone, and sort sequence. Any bins added since the last process run are added to the list. Any bins deleted since the last process run are removed from the list. The bin additions/deletions do not affect the nightly processing routine since it keeps track of where it is in the list and picks up from that point the next time it is run.
Cycle Count Nightly Processing
Path: Inventory > Physical Inventory > Cycle Counting Procedure > Cycle Count Nightly Processing (/CYC.COUNT.NIGHTLY.PROCESS)
The main purpose of the nightly processing routine is to populate a bin count queue record with the bins to count in a given day. The bin count queue record is sorted by creation date in the actual counting process to assure a first in / first out counting methodology. A statistical measure of the number of times scheduled is updated for each zone that has a bin scheduled in the daily count. This is to assure that a zone is not scheduled for counting more times than its allotted bin count frequency in a given year.
When this process is run, the system performs the following:
1
First it checks the status of each frequency in Cycle Count Control Maintenance to determine if it is included in the daily cycle count.
2
Next it determines for each frequency, the number of bins that should be counted for that day (the number of daily count bins per frequency calculated by Monthly Processing).
3
The determined number of bins for each frequency is then obtained from the list of bins for each frequency.
For example: If monthly processing determines for frequency 6 that 3 bins should be counted each time, then nightly processing would determine the last bin selected and take the next 3 in the list for the next count cycle. This routine then updates a control file with the last bin selected for each count frequency. Once all bins in the list are counted, the system updates for each zone in the frequency, the number of times the zone was scheduled. If the zones have not been counted the maximum number of times for the year, the system starts over from the beginning of the list.

First Level Queue Report
Path: Inventory > Physical Inventory > Cycle Counting Procedure > First Level Queue Report (/CYC.COUNT.FIRST.LEVEL.RPT)
This report shows bins to be counted from the bin count queue and the number of product counts per bin to be counted.
•
Division Number: Optional field used to define a division for the first level queue report. If left blank, all divisions are selected.
•
Warehouse Number: Mandatory field. Defines the warehouse for the report.
•
Zone: Optional field used to define the zone where the count is to be performed.
•
Open or Assigned: Determines whether assigned and/or non-assigned bins are printed on the report.
-
O - The system prints a list of only those bins in the first level big queue that have not yet been assigned to a cycle counter.
-
A - The system generates the list only for assigned bins.
-
If the field is left blank both open and assigned bins are included.
Once the process is executed, the system generates the first level queue report. The report displays zones, zone descriptions, bin numbers, bin descriptions, counter assignments (for those bins that were assigned), count date, and the number of counts per bin.
Cycle Count Entry (1st Level)
Path: Inventory > Physical Inventory > Cycle Counting Procedure > Radio Frequency Cycle Count Entry (1st Level) (/RCCE)
Warehousing > Handheld RF Menu > Physical > Cyc Cnt Ent. 1st (/RCCE)
Warehousing > Truck Mount RF Menu > Physical > Cyc Cnt Ent. 1st (/TRCCE)
Path: Inventory > Physical Inventory > Cycle Counting Procedure > Cycle Count Entry (1st Level) (/CYCE)
This entry program is used to scan products and enter actual counts.
•
Warehouse Number: Specifies the warehouse for entering the cycle count results.
•
Counter ID: The person who performed this count.
•
Zone: The zone where this count was performed.
•
# Of Bins: The number of bins this person counted.
Once these fields are filled in, the following screen displays:
•
Zone: Displays the zone specified on the previous screen.
•
Bin: Displays the first bin ID in the specified zone that needs to be counted.
•
Product Number: Displays the product number when the barcoded product is scanned. In versions 17 and higher, the product numbers are sorted in alphanumeric order for each bin location.
•
Lot: Prompts for a lot number if the product is lot controlled. If the product is not lot-controlled, this field is skipped.
•
Quantity: Prompts for the quantity counted for this product/bin/lot combination. The system compares the quantity entered (which is the actual count of the product) with the system onhand at that time. If no discrepancy exist, the system redisplays the screen for the next product number.
If there is a discrepancy between actual count and the system onhand, the system checks the cost of the item in the cost/tolerance maintenance record. If the count discrepancy percentage is less than or equal to the tolerance percentage in this file, the system performs an automatic inventory adjustment on the product when the current bin is updated. The system uses the default reason code defined in the Inventory Option Parameter Maintenance for this automatic inventory adjustment.
If the count discrepancy is not in tolerance, the user is warned and given a chance to recount. (This check is performed until the count is in the tolerance or until the user elects not to recount.) If the counter elects to recount, then the system returns to the Quantity field so a new quantity can be entered. If the counter elects not to recount, the item is inserted into the cycle count second level queue for further analysis when the current bin is updated. The current bin onhand is adjusted up/down to match the count quantity entered and a fictitious bin called Cycle is adjusted up/down the opposite way. The result of this transaction is a behind the scenes bin movement (between the count bin and the fictitious Cycle bin). No inventory adjustment is performed at this time. Any needed inventory adjustment will be done with the second level queue count.
•
F2-Upd Bin: Saves the information entered. The system redisplays the screen, prompting for the next product number. Repeat the procedure explained above to enter the appropriate product number, lot number, and quantity for the specified bin.
•
F6-Exit: Exits the screen without updating the product. Note: if quantities have already been entered, the system will not save or update these quantities. To exit and save the quantities, use the F2-Update Bin key. The message 5 Product/Lot Combinations Have Not Been Counted! Do You Wish To Continue? displays. When you respond ‘Yes to this prompt the message All Assigned Bins Were Not Counted. Uncounted Bins Will Be Returned To Queue! displays letting you know that the session was not completed and that the remaining items were returned to the cycle count queue to be counted.

Second Level Queue Report
Menu Path: Inventory > Physical Inventory > Cycle Counting Procedure > Second Level Queue Report (CYC.COUNT.SECOND.LEVEL.RPT)

Entering Second-Level Counts
Menu Path: Inventory > Physical Inventory > Cycle Counting Procedure > Radio Frequency Cycle Count Entry (2nd Level)
Warehousing > Handheld RF Menu > Physical > Cyc Cnt Ent. 2nd (RCCS)
Warehousing > Truck Mount RF Menu > Physical > Cyc Cnt Ent. 2nd (/TRCCS)
This step is performed whenever a bin count from the first level count does not match the onhand quantity for that product and the difference between the actual count and the onhand quantity is greater than the tolerance allowed on the cost/tolerance maintenance record. During nightly processing, the system creates a bin queue count record for the actual counting process which is used with this entry program. This entry program cycles through all the product/bin/lot combinations until all products have been covered. Products with discrepancies should be recounted prior to this entry.
•
Warehouse: This field is mandatory and defines the warehouse number for the second level counts.
•
# Products: This field can be used to enter the number of products to be displayed on the entry screen for the cycle counter. For example, if # Products = 3, three products appear when you press F3 to search. If the # Products field is left blank, all the products available for cycle counting for the warehouse number entered display.
•
Counter ID: The person who performed the count.
•
Product Number: Defines the product number for the second level count quantity. If the Default P/N in 2nd Level Entry flag in Warehouse Maintenance is set to Y, the product number is defaulted in the Product Number field.
•
Bin: Displays the bin number for the product specified.
•
P: Displays an asterisk to denote a first level problem count.
•
C: This field is blank until after the second level count has been entered. Then an asterisk is displayed.
•
Qty: Displays the count quantity entered during the first level count.
•
U/M: Displays the unit of measure for the product number entered.
•
F2-Upd: Updates the count after the correct quantity is entered.
•
F6-Exit: This function key is used to exit without updating the bins.
When the product number has been entered and the Bin field is accepted, another screen displays to enter the second level count.
The system automatically displays the bin and product number as well as the count quantity and the discrepancy quantity from the First Level Cycle Count Entry.
•
Count: Enter the second level count.
The prior summary screen comes back displaying the next bin for the current product - if there is one. Repeat this process for each bin with an asterisk in the ‘P’ field. If all problem bins are not counted, no further processing is done. Bins with no discrepancies for the warehouse/product combination do not have to be recounted but may be counted to reflect accurate counts among all of a product’s bins.
Once all bin counts for the product are entered, press <F2> to update. A dialog box displays showing the quantity and extended value of the inventory adjustment that is about to take place. If you elect to proceed, the system performs an automatic inventory adjustment behind the scenes to bring the product’s onhand quantity in line with the count quantity. The next product with a discrepancy may then be entered.
Lot Products: The only difference with lot-controlled products is that the screen used to correct the count displays a P field beside the Lot field. This is used to define the lots that have a problem.
After the count is entered, the system displays the next lot number for the product. All lots are cycled through. Press F2-Upd to return to the first screen. If the product has another bin, that bin is displayed.

Emergency Cycle Counts
Emergency cycle counts are created when the system detects a potential error (e.g. negative bin quantity, change during picking).
Emergency Cycle Count Entry
Menu Path: Warehousing > Processes > Emergency Cycle Count Entry (ECCE)
This allows a product and bin to be identified to put on the emergency cycle count queue.
RF Emergency Cycle Count Entry
Menu path: Warehousing > Handheld RF Menu > Processes > Emg Cycle Cnt Ent (/RECCE)
Warehousing >Truck Mount RF Menu > Processes > Emg Cycle Cnt Ent (TRECCE)
These processes identify a product and bin to put on the emergency cycle count queue using RF devices.
Emergency Cycle Count Inquiry
Menu Path: Inventory > Inquiries > Emergency Cycle Count Inquiry (/ECCI)
This inquiry is used to monitor any product picking adjustments done throughout the day. If a transaction was short picked for example, the product automatically updates this inquiry. The detail includes the product, bin location short picked, the user and the transaction (ex: sales order number).
Physical Inventory - RF/Tags
Menu path: Warehousing > Handheld RF Menu > Physical > Phys.Tag Ent (/RF.PHYS.TAG.ENT)
RF physical tag entry is used to enter counts and generate tag numbers during a physical inventory only if you will use RF to count. You do not need to print physical tags. All other physical inventory tag processing is the same. The RF unit replaces the manual physical tag printing and physical entry steps. You can enter a tag number (which is always new), product number, bin and count. These entries are updated as manual tags in the Physical Inventory Tag Procedure.

General Ledger
Journal Entries
Journal Entry
This process was modified to work with the Bal G/L Posting By Location option parameter. If the option parameter is set to Y, the system will verify that the entry balances by location upon exiting. If it does not, the system will prompt for a password. Either the entry must balance by location or a valid password must be entered to exit a journal entry. This option uses the password process called ONE.SIDE.PASSWORD. If this password process is not set up then you will not be allowed to exit the journal entry without a balanced entry. If the option parameter is set to N, the system will work as it did in the past by checking that the debits and credits are balanced. If not, you are warned and prompted for a password. The system will not allow you to continue if a password is not set up unless the journal entry is balanced. See System Option Parameter Maintenance for more information.
G/L Balances Posting
•
Job Accounting is a new option used for posting the job accounting module. This module will post both sides of the job accounting processing; the Sales Order Invoice Processing which does the inventory relief, and the job billing which posts the transaction to the customer. The job accounting is also listed as a separate module in the inquiries with a code of JA. The drill down inquiries will access new job history inquiries in place of the standard order history inquiries. See General Information Job Accounting for more information.
•
The balancing posting processes were modified to no longer stop posting all items when certain errors are found with a transaction. Instead it updates an audit file, marks the transaction, and continues posting the rest of the transactions in the module. The errors that this works with are Multi-Value Mismatch, Debits/Credits Out of Balance, Missing Account Number, and Location Out of Balance if the option parameter is set to Y. If an error other than those mentioned is found then the system will stop posting for the module and notify the user of the problem. The Unposted G/L Error report will list transaction information for reconciliation purposes. See System Option Parameter Maintenance and General Ledger Reports for more information.
•
This process was modified to work with the Bal G/L Posting By Location option parameter. If the option parameter is set to Y, the system will verify that the transaction balances by location. If not then it marks the transaction, updates the audit file, and continues to the next transaction. A report may be run for these transactions. If the option parameter is set to N, the system will just check (as it did previously) that the debits and credits are balanced. See System Option Parameter Maintenance and General Ledger Reports for more information.

This process was modified to work with the Multiple Fiscal Periods flag in Company Maintenance. G/L Balances Postings will, behind the scenes, post to the correct fiscal period on the location’s fiscal account. All postings will be done from the live account. The posted fiscal period for the fiscal account is determined by the transaction’s fiscal period on the live account instead of the transaction date.
The system will determine if the Multiple Fiscal Periods flag is set to Y and if so will locate the Separate Fiscal value for the transaction’s location. If one is found, the system will next determine the fiscal period to post to on the fiscal account. This is accomplished by determining the fiscal period on the live account and then checking the cross-reference fiscal period table for the fiscal account to determine the corresponding fiscal period. If that fiscal period is closed, the system will determine the next available fiscal period to post to. The system then posts the transaction to the live account and copies the transaction to the appropriate fiscal account with the correct fiscal period for that fiscal account. For example, the Separate Fiscal field for location 10 is set to MAR. A transaction for warehouse 10 with a fiscal period of 99-05 will post on the live account to fiscal period 99-05 (if open) and a copy of the transaction is made in the MAR fiscal account with a fiscal period of 99-08.
The actual posting on the fiscal account will depend on which system platform you are on. G/L Balances Postings when ran on the live account will copy the transactions to the correct fiscal account. Pick clients will need to log to each fiscal account and run the G/L Balances Postings. For Unidata clients, when the G/L Balances Postings are run on the live account, the system will spawn off phantom processes that run the G/L Balances Postings on the fiscal accounts. Prelude will need to set up these phantom processes for you. For both platforms, you will need to log to the fiscal accounts to close a period and/or run fiscal reports.
See System Company Maintenance, System Location Maintenance, and System Mult-Fisc Period Table Maintenance for more information.
Financial Report Printing
Report in Native is a new prompt that allows you to define if the report should print in native currency. This prompt is only available if multiple currency is turned on. If this flag is set to Y, the system will display all accounts in native currency. If this flag is set to N, the system will display all accounts in their own currency. For example, US accounts will display in dollars, English accounts will display in pounds, Japanese accounts will display in Yen, and etc.
End of Year Closing
This process was modified to work with multiple Retained Earnings accounts. The General Ledger End of Year Closing process closes out all income and expense accounts to the retained earnings account defined in the General Ledger Table. In other words, it creates journal entries to zero out all income and expense accounts and posts the balance of each account to the retained earnings account(s). The journal entry for the retained earnings posting uses a fiscal period of XX-YE where XX is the current year (I.E. 99-YE). If a wildcard is used in the prefix of the Retained Earnings account in the G/L table, the number of retained earnings accounts updated is equivalent to the number of different prefix combinations for the income and expense accounts.
Examples
If wildcards are used, each income and expense account is posted to it own prefix. For example, if the G/L table accounts are set up as follows:
Retained Earnings 01 XX 01 9109
Income / Expense 1 01 10 00 1234 balance = 50.00
Income / Expense 2 01 20 00 5678 balance = 10.00
Then the year end procedure would post to the following retained earnings accounts:
50.00 into account 01 10 01 9109
and
10.00 into account 01 20 01 9109
If wildcards are not used, all income and expense accounts are posted to the retained earnings account defined in the G/L table. For example, if the G/L table accounts are set up as follows:
Retained Earnings 01 10 01 9109
Income / Expense 1 01 10 00 1234 balance = 50.00
Income / Expense 2 01 20 00 5678 balance = 60.00
Then the year end procedure would post to the following retained earnings account:
110.00 into account 01 10 01 9109
When the General Ledger End of Year Closing procedure is ran and the Retained Earnings account is set up in the G/L Table with wildcards, the system will verify that all the appropriate Retained Earnings accounts are set up in Chart of Accounts Maintenance. If some are missing, a message (similar to the one below) is displayed informing you of the missing accounts that need to be set up before this procedure can be completed.
Once these retained earnings accounts are established in Chart of Accounts Maintenance, the general ledger end of year closing process can be performed to update the appropriate retained earnings accounts. See General Ledger Table Maintenance for more information.
Maintenance
Chart Of Accounts
This procedure was modified to work with the Multiple Fiscal Periods flag in Company Maintenance. When Multiple Fiscal Periods is turned on and if the location points to a fiscal account, the system will automatically update the location’s fiscal account when a chart of account is added or changed in the live account. For example, the Separate Fiscal field for location 10 is set to MAR. When adding account 5845 for Work Order Expense in the live account for location 10, the same account is also created in the MAR account. Chart of Accounts should be maintained in the live account instead of in the fiscal accounts. See System Company Maintenance and System Location Maintenance for more information.
Chart Of Accounts Description
This procedure was modified to work with the Multiple Fiscal Periods flag in Company Maintenance. When Multiple Fiscal Periods is turned, the system will automatically update each fiscal account when an item is added or changed in the live account. For example, the description for account 4100 is changed from DIRECT SALES to SALES - DIRECT. When this screen is saved and exited, account 4100 in each fiscal account will be updated with the new description. When a new account description is created, the new account description will be copied behind the scenes to each fiscal account. Chart of Accounts descriptions should be maintained in the live account instead of in the fiscal accounts. See System Company Maintenance for more information.
Chart Of Accounts Duplication
This procedure was modified to work with the Multiple Fiscal Periods flag in Company Maintenance. When Multiple Fiscal Periods is turned on and if the To location points to a fiscal account, the system will automatically update the To location’s fiscal account when G/L accounts are duplicated in the live account. For example, the Separate Fiscal field for location 30 is set to OCT. When copying prefix 01!!10!00 to 02!!30!00 in the live account, the accounts are also created in the OCT account. Chart of Accounts Duplication should be run in the live account and not in the fiscal accounts. See System Company Maintenance and System Location Maintenance for more information.
General Ledger Table Maintenance
•
Core Receivable is a new account that is debited for the cost of the core when a sales order with an exchange item but not the core item is invoiced. This is a situation where the Auto Core Charge option parameter is set to N and the Bill Core Charge prompt in sales order entry was answered No. This account is credited for the cost in the pending core record when a core item is billed against a pending core record in Sales Order Entry, when a core item is returned in Inventory Adjustment Entry against a sales order (pending core record), when a core bank item is applied against a pending core record in Core Bank Issues Entry or in the Automatic Core Application Process, or when a pending core is written off in Core Bank Issues Entry. See Order Processing Sales Order Entry, Inventory - Inventory Adjustment Entry, Inventory Processes Core Bank Issues Entry, and Inventory Processes Automatic Core Application Process for more information.
•
Inventory Warranty In Transit is a new account that is used to hold the value of warranty inventory in transit and takes the place of the Inventory In Transit To Location account that is used for normal transfers. This account is debited in the shipping location when a type 03 transfer is shipped in Transfer Shipment Entry and credited in the shipping location when a type 03 transfer is received in Transfer Receipts Entry. See Inventory Stock Movements for more information.
•
A/P Not Invoiced – Inventory can now be overridden at the product line level. See Inventory Product Line Maintenance for more information.
•
Core Bank Payable is a new account that is credited for the cost of the core when a core item is placed into the core bank in Inventory Adjustment Entry. This account is debited for the cost in the core bank record when a core bank item is applied in sales order entry or when a core bank item is applied against a pending core record in Core Bank Issues Entry or in the Automatic Core Application Process. See Inventory - Inventory Adjustment Entry, Order Processing Sales Order Entry, Inventory Processes Core Bank Issues Entry, and Inventory Processes Automatic Core Application Process for more information.
•
Customer Job Use Tax Payable is a new account that is credited from job invoice processing when a line item is flagged as use tax instead of sales tax. See General Information Job Accounting for more information.
•
Customer Job Use Tax Cost is a new account that is debited from job invoice processing when a line item is flagged as use tax instead of sales tax. See General Information Job Accounting for more information.
•
Transfer Cost Of Goods is a new account that is debited in the shipping location when a Transfer Revenue Matrix is maintained between warehouses and product is transferred between two locations. The amount posted is the cost of the product on the transfer extended by the quantity received in Transfer Receipts Entry. See Inventory Misc Code File Maintenance for more information.
•
Transfer Revenue is a new account that is credited in the shipping location when a Transfer Revenue Matrix is maintained between warehouses and product is transferred between two locations. The amount posted is the cost of the product on the transfer multiplied by the matrix markup extended by the quantity received in Transfer Receipts Entry. See Inventory Misc Code File Maintenance for more information.
•
Retained Earnings was modified to allow for a wildcard division, region, location, department, and/or cost center. This allows a posting to different Retained Earnings accounts based on G/L prefixes instead of to just one account. When using wildcards, a retained earnings account must be established in Chart of Accounts Maintenance for each possible combination. You are warned in the Year End process if an account is not established for a combination. See Year End Closing Process for more information.
•
Sales Customer Job is a new account used with the Job Accounting module. Credits are posted to this account when sales are processed through the Job Billing process. See General Information Job Accounting for more information.
•
Core Cost Variance is a new account that is debited for the cost of the pending core record when a pending core is written off in Core Bank Issues Entry. This account is posted to for the difference between the cost in the pending core record and the cost in the core bank record when a core banked item is applied to a pending core record in Core Bank Issues Entry or in the Automatic Core Application Process. This account is also posted to when the applied core bank quantity does not match the pending core quantity in Sales Order Entry, Inventory Adjustment Entry, and Core Bank Issues Entry. See Order Processing Sales Order Entry, Inventory - Inventory Adjustment Entry, Inventory Processes Core Bank Issues Entry, and Inventory Processes Automatic Core Application Process for more information.
•
MFGR Rep Commission Due is a new account that is posted to when a type 17 Manufacturer Rep sales order is invoiced. The gross profit of the sale is posted to this account. This account is overridden if an account is specified in Product Line Maintenance. See Order Processing Sales Order Entry for more information.
•
MFGR Rep Sales is a new account that is posted to when a type 17 Manufacturer Rep sales order is invoiced. The amount of the sale minus the commission is posted to this account. This account is overridden if an account is specified in Product Line Maintenance. See Order Processing Sales Order Entry for more information.
•
MFGR Rep Cost of Sales is a new account that is posted to when a type 17 Manufacturer Rep sales order is invoiced. The total merchandise amount of the sale is posted to this account. This account is overridden if an account is specified in Product Line Maintenance. See Order Processing Sales Order Entry for more information.
•
Customer Job Work In Process is a new account used with the Job Accounting module. Debits are posted to this account during invoice processing for sales orders with a job number whose Change Order Control flag is set to Y. This account is credited during the Job Billing process. See General Information Job Accounting for more information.
Reports
Unposted G/L Error Report is a new report that displays all transactions that have failed the balance posting routine due to a location out of balance, a multi-value mismatch, a missing account number, and/or debits/credits out of balance. This report is mainly for Prelude to use to fix problems postings. This report will display the posting error(s), account numbers, debits, credits, and the out of balance amount for ease of correcting the problem. See General Ledger G/L Balances Posting for more information.

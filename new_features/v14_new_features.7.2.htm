Purchasing and transfer information is now saved when a sales order is placed on credit hold or on administrative hold or when the order is exited without generating the purchase order or transfer. Previously, when a sales order set up for automatic purchase order and/or transfer generation was placed on hold, the purchasing and transfer information was discarded. Therefore, when the order was released from hold, the user had to re-enter the order and walk through each line to set them up again for purchasing or transfer. This process was modified so that when a sales order set up for automatic purchase order and/or transfer generation is placed on hold, the user is prompted to retain or discard the purchasing and/or transfer information. If the information is retained, the user will only have to re-access the sales order. They will not have to walk through each line item.
Previously, when a sales order set up for automatic purchase order and/or transfer generation was exited without generating the purchase order or transfer, the purchasing and transfer information was discarded. This process was modified to retain that information. Therefore, the user only has to re-access the sales order. They will not have to walk through each line item.
When an order that has saved purchasing/transfer information is access, you are warned that auto purchasing/transfer requests exist and allowed to retain the information or discard it. If you elect to retain the information, the original purchasing/ transfer information is used for automatic purchase order and/or transfer generation. If you elect to discard the information, each line item must be reprocessed to generate the purchase order and/or transfer needed to fill the order.

There is a new Order Type of 16 which is used for order projections. Products entered on this type order will be completely backordered and will never automatically allocate. The user will have to manually allocate inventory to the order. Basically, this order is a projection of what the customer will order of certain products throughout the year. As the customer actually orders the items, the quantity is decreased on the projected order and the items are placed on a regular order. The system will not allocate to this order type. The products on this type order will have to be manually allocated meaning that the user will have to force a pick quantity for each line item. See System Maintenance Order Processing Controls for more information.
•
There is a new Order Type 17 for Manufacturer Rep orders. This type order is used in situations where the customer orders direct from the vendor and receives the material from them but you receive commission from that sale. The manufacturer rep order is entered to recognize the request. This type order is processed similar to a type 04 direct order except this type order only updates sales information; customer sales and sales analysis. It is handled similar to a direct order in that no inventory commitments are made for the items on the order but the order will still appear in Inventory Inquiry until the order is billed. Unlike a direct order, accounts receivable is not affected by this type order. Special manufacturer rep accounts must be established in the General Ledger Table before utilizing this feature. This is also known as a Letter Of Credit order.
Any product entered on this type order is processed like a service item since no inventory commitment is made This type order will skip calculating taxes and the other charges on the completion screen. The order may only be suspended or completed for invoicing. When the order is completed and invoiced, the general ledger should reflect the commission due calculated as the gross profit of the sale.
See System Maintenance Order Processing Controls, General Ledger G/L Table Maintenance, and Order Processing Reports for more information.

This screen was modified to work with the Auto Core Charge option parameter. If the Auto Core Charge option parameter is set to Y, the core item is automatically placed on the order for billing when an exchange item is entered. If the Auto Core Charge option parameter is set to N, you are prompted to bill the core charge, not bill the core charge, or to apply a core bank item when an exchange item is entered. If Yes is chosen, the core product is added to the sales order. If No is chosen, the core product is not added to the order but you are prompted for the Due Date. This is the date the core should be returned by. When the order is invoiced, a PENDING.CORES record will be created if the Keep Pending Cores file update parameter is set to Y. If Core Bank Credits is chosen, the system will display the customer’s core bank records and prompts you for the core bank item to apply. The system will not accept the core bank option if the customer does not have any banked cores. Note: this process allows the quantity applied from the core bank file to be different than the line item quantity. For example, if the line item is for a quantity of one but the core bank record contains a quantity of two then the two cores are applied against the one unit. The cost difference would be posted to the Core Cost Variance account. See System Maintenance File Update Parameters, System Maintenance Order Processing Option Parameters, Inventory - Inventory Adjustment Entry, Inventory Processes, General Ledger G/L Table Maintenance, Inventory Misc Code File Maintenance, Order Processing Miscellaneous Reports, Inventory - Inventory Adjustment Reports, Order Processing Inquiries, Inventory Core Inquiries, System Processes Purge Utility Menu, and System Processes Rebuild Menu for more information. This process was modified to perform automatic product substitution for customers who accept substitutions. If the customer’s Accept Substitutes flag is set to Y, one of the following steps will occur when a product that has substitutes defined is entered in sales order entry. See Inventory Product Maintenance for more information.
-
If the original product’s Product Substitution Type is set to B, the system will automatically replace the original product with the substitute product even if the original product has available quantity. Pricing will be based on the substitute product.
-
If the original product’s Product Substitution Type is set to A or P then the system will determine if the quantity available for the original product is greater than 1.
If it is then you are taken into the line item detail screen. When the line item detail screen is exited, the system will determine if there is a backordered quantity.
If there is, the system will automatically replace the original product with the first substitute product with enough available inventory to meet the requirement. If one is not found, no substitution will take place and the original product will remain on the order. Pricing for the line item will depend on the substitution type. If the substitution type is A then pricing will be based on the substitute product. If the substitution type is P then pricing will be based on the original product.
If there is not a backordered quantity then the line item remains as is.
If the quantity available for the original product is less than or equal to 1, then the system will automatically replace the original product with the first substitute product with available inventory before taking you into the line item detail screen. Note: the replacing product may or may not have enough available inventory to meet the need. Pricing for the line item will depend on the substitution type. If the substitution type is A then pricing will be based on the substitute product. If the substitution type is P then pricing will be based on the original product.
•
If the original product’s Product Substitution Type is set to S or is null then you are taken into the line item detail screen where Sub is displayed to let you know the product has substitutes. You must manually enter the F6-Subs screen to choose the substitute product.
If there is a substitution and if the original product contains substitution comments, those comments are added to the order when the line item detail screen is exited.

-
If the customer has one or more account code patterns, then the account code entered is checked against these patterns.
-
If the entered account code does not match any of the identified patterns, and the customer has a master customer assigned which has one or more account code patterns, then the account code is check against the account code patterns in the master customer record.
-
If a pattern match is not found in either of the above, then the user is informed that a match has not been made against the valid account code patterns for that customer.
-
If the customer’s Requires Acct Code flag is set to Y, then the you are forced back to the Customer Acct Code field until a valid match can be made against the account code patterns.

The tax flags for the freight and additional charges fields were removed from this screen and placed on the F5-Tax Jur > F5-Tax Flags screen. Require Acct Code is a new field that determines if an accounting code must be entered for each sales order or quote line item for the customer. If this flag is set to N, then an accounting code may be entered for line items but it is not required. If this flag is set to Y, all line items on sales orders and quotes for the customer will require an accounting code to be entered. When exiting the line item detail screen, the system will check for an Accounting Code. If the line item does not contain one, a warning is displayed and you are automatically taken into the additional information screen to enter the code. You may not continue till an accounting code is entered. This logic only works if the line item detail screen in SOE and QOE is accessed. You could, however, manually access each line item and enter the accounting code.
You also have the ability to enter a pattern (similar to P/O Number logic) to match the accounting codes. This insures that a customers accounting code is entered correctly in sales order entry and quote order entry by checking a list of customer specified account code patterns for a match. Installing patterns in a customer’s record informs the system that any customer account code entered in sales order entry or quote order entry must be validated as a correct pattern. The Require Acct Code field must be set to Y to access the screen where these codes are entered. After that you may turn the field off if the customer does not require an accounting code on every order but still perform pattern matching should the operator enter an accounting code. This means that pattern matching can be performed with or without requiring the accounting code to be entered. Patterns may include alpha, numeric, both alpha and number, dashes, and strings. Though unlimited patterns may be entered, only one pattern should be entered per line. Patterns may be entered in using the standard PICK pattern matching methodology. Amounts paid after this number of days will not be eligible for a rebate. See Accounts Receivable Cash Receipts Entry for more information.
•
Exclusion Type is a new field that defines the product group that will be excluded from the current rebate program. Options are G for Product Price Group, L for Product Line, P for Product, and V for Vendor. This field and the Exclusion List field are used when there are products that are not eligible for a rebate. The combination of these two fields defines those products that are not eligible for rebates. See Invoice Processing for more information.
•
Exclusion List is a new field that defines which products are excluded from the rebate program. Enter Vendor Number, Product Line, Product Price Group Number, or Product Number depending on how the Exclusion Type field is set. Multiple entries may be made. F3-Lookup and validation is provided per Exclusion Type. In other words, if the type is set to Vendor, then the F3-Lookup and validation is on the Vendor file. See Invoice Processing for more information.
Order Entry Control Files - Override Shipping Whse Matrix
Added a new matrix type of 3 for Vendor/Selling Whse Override which allows you to override the default shipping warehouse for a vendor/selling warehouse combination. The system looks at a product’s primary vendor to determine if it is applicable for this matrix type. The selling warehouse is pulled from the sales order. All products for the vendor sold from the defined selling warehouse will default to the defined shipping warehouse.

F6-Commission Cost is a new function key used to define a cost to use for commissions when the corresponding matrix is used to price a line item in Sales Order Entry. Commission cost for the line item is calculated based upon the Cost Method. Options are * for percentage of a price/cost field, $ for a cost, or T for table. Miscellaneous Reports
•
The Customer Rebate Detail Report is a new report that prints customer invoices eligible for a rebate for a given period of time and for a Credit Report Freq. User is prompted for a customer number, rebate program number, rebate period (Credit Report Freq), and a range of invoice dates. The rebate period and date range must be entered. User is also prompted for a report description that will print on the report. This allows you to define in more detail, what the report is for. This does not replace the report description but allows you to add to it. For example, the standard report description is Customer Rebate Detail Report. You could set this field to IHS Rebates – 1998. This would print on the fourth line of the header. This report displays for each invoice the invoice date, the invoice amount, the total merchandise amount, the amount eligible for a rebate, the amount paid in the Eligible Days to Pay days from the Invoice Date, the amount paid after the Eligible Days to Pay days from the Invoice Date has passed, the unpaid balance, and the rebate amount. This report sorts and breaks by customer name. See Order Processing Invoice Processing and Accounts Receivable Cash Receipts Entry for more information.

Core Reports is a new option to this screen.

Pending Cores Due Report is a new report that displays pending core records created because an exchange product was invoiced and the Bill Core Charge flag was set to N. You are prompted for a beginning and ending customer number, warehouse, and a core due date. Any record with a due date less than or equal to this date would be displayed. This report breaks and totals by warehouse by customer. See Sales Order Entry Line Item Level and System Maintenance Order Processing Option Parameters for more information.
•
Core Returns Due Aging Report is a new report that displays pending cores that have not been received yet. It gives the user an idea as to what they have remaining to be returned and when it is expected to come in. This report prompts for a customer range, As Of date, and/or Core Code. If a customer range is not entered then all customers will be selected. The report displays when the cores are due or how long they are past due based on the due date relative to the entered As Of date. In other words, the data is put in the appropriate columns based on the As of Date entered at run time and the Due Date of the cores. For example, if a pending core record’s Due Date is 9/1/99 and you enter an As Of Date of 7/15/99 then the item would appear in the Future 31-60 bucket. But if you enter an As Of Date of 8/15/99 then the item would appear in the Current bucket. Likewise, if 10/15/99 were entered, the item would appear in the Past Due 31-60 bucket. This report sorts by customer by core type code by product number and page breaks on customer. See Sales Order Entry Line Item Level for more information.
•
Core (Issues) Sold on Exchange Report is a new report that displays core items sold on exchange. In other words, this report displays pending cores records created because an exchange product was invoiced and the Bill Core Charge flag was set to N. You are prompted for warehouse, customer number, and an invoice date range. This report will display for each customer the invoice number, invoice date, order number, and ship date. For each invoice, the exchange product number, core product number, ship quantity, core charge, quantity applied, last applied date, quantity due, and the extended charges are displayed. This report sorts and breaks by customer by invoice number and totals on invoice number and customer number. See Sales Order Entry Line Item Level for more information.
